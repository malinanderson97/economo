**IS-LM-PC Teaching Tool — Master Plan**

*Consolidation + progressive-disclosure build plan · 18 June 2026*

# 1. Purpose

The full model (IS-LM-PC, Okun-modified, both closed and open economy) is built and equation-matched to Blanchard. The next phase is a teaching version that reveals the model progressively — every block stays visible, but blocks not yet taught are greyed out and locked. This document is the single plan for that phase and supersedes the older scattered roadmap docs.

# 2. Two design axes

Keeping these separate is what stops the tool drifting back toward 'demonstrate everything.'

- **Breadth (lesson level): **which blocks are live vs greyed-and-locked. The full model is always on screen so the learner sees where they are heading.

- **Depth (drill-down): **inside any live block, click a curve to expand its derivation — e.g. IS-LM unfolds into the IS relation + the LM-from-money-market construction. Available regardless of level.

# 3. Block structure (the onboarding sequence)

*Single progressive learning tool: one model that the learner unlocks block-by-block from closed short-run through to the full open-economy medium run. There is no longer a separate "free/closed" and "paid/open" product — see §7. The full model is always on screen; blocks not yet taught are greyed and locked.*

**Settled five-block order (confirmed by Frank): GOODS → ISLM → UIP → PC → DEBT.**

| **Block** | **Live (unlocked)** | **Free params** | **Blanchard anchor** | **Rate / inflation regime** |
| --- | --- | --- | --- | --- |
| **GOODS** | Keynesian cross (Y=Z) | G, T, c₁ | Ch. 3, eq. 3.6; multiplier 1/(1−c₁) | no interest rate yet |
| **ISLM** | + IS-LM chart, money market, policy rate | + i | Ch. 5; eq. 9.1 (IS) | **NOMINAL rate only: r = i.** Prices/inflation constant (Ch. 5 core short run). NO πᵉ control, NO Fisher line, NO real-rate line. |
| **UIP** | + imports/exports, UIP, exchange rate | + m₁, x₁, ε, Y*, i* | Ch. 18 (eq. 18.4); Ch. 19 (UIP eq. 19.5) | still nominal r = i (UIP is an open-economy IS extension, not the medium-run expansion) |
| **PC** | + Phillips curve, expectations, Yₙ, Okun | + α, z, θ, credibility, **πᵉ** | Ch. 8 (eq. 8.4–8.9); Ch. 9 | **NOMINAL→REAL distinction switches on: r = i − πᵉ.** Fisher line, real-rate line, and the πᵉ control all appear here together (medium-run/expectations expansion). Time-controls (Step/Reverse/Rebalance) also gate here. |
| **DEBT** | + government debt dynamics | + B, g | Ch. 22 (debt dynamics) | full model live |
| *(later) Forward FX (v2)* | + forward-looking FX expectations | + expected future ε | Ch. 20 (saddle path) | out of current scope per README |

**Critical engine consequence (DECISION OVERRIDE — supersedes earlier "πᵉ stays in ISLM"):** πᵉ unlocks WITH the PC block, not in ISLM. The ISLM (and UIP) stages are the Ch. 5 core short run using the nominal rate with r = i, where the nominal/real distinction does not yet exist. This is implemented (item A, §9 Done): solve() computes r = i when PC is locked and r = i − πᵉ when unlocked, gated on tutorial state via effectivePiE(s); verify_v19 #16 pins output invariant to πᵉ when PC locked and responsive when unlocked. The design decision reverses a previously locked call ("πᵉ stays in ISLM") and is still to be confirmed with Frank, but the engine work is done and committed.

# 7. Product framing & adoption

**The product is now a single progressive learning tool**, not a free/paid two-tier split. The earlier plan had a free closed-economy model and a paid open-economy model; that is superseded. The full model (closed → open, GOODS through DEBT) lives in one tool, revealed block-by-block via the onboarding layer. Any remaining "free tier / paid tier / v19 paid" language in the code (e.g. the Copy-state export header), the Correspondence doc, or the README is stale and tracked for cleanup in §9.

The adoption thinking below still holds, just without the tiered-pricing mechanics:

- **Individual lecturer / convenor is the realistic first adopter** — high autonomy, low budget. Frank using it in his own lectures is the single best adoption lever.
- **Department next** — a few people who know each other, modest teaching-resource budget.
- **University central / publishers are later, slower routes** — committee procurement, accessibility/security review; only worth it once the tool is proven in a real module.
- **Adoption inertia, not competitors, is the real enemy.** Lecturers reuse materials for years; a personal introduction beats any feature.
- **The realistic first loop:** one friendly adopter uses it with real students for a term → collect a short endorsement + student reaction → use that to approach the next two or three convenors. Turn the loop once and you have proof; out of scope to formalise pricing before the first yes.
- **Don't over-invest before the first yes** — tie development to actual adopter requests, not speculative features.
